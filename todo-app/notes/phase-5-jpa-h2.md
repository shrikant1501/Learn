## The Four-Level Stack

```
Your Code  (TodoRepository, TodoServiceImpl)
    ‚Üì
Spring Data JPA    ‚Üê generates repository implementations at runtime
    ‚Üì
JPA (specification) ‚Üê defines how Java objects map to DB tables
    ‚Üì
Hibernate          ‚Üê the actual ORM implementation of JPA
    ‚Üì
H2 Database        ‚Üê the relational database engine (in-memory)
```

- JPA = the spec (like the `List` interface)
- Hibernate = the implementation (like `ArrayList`)
- Spring Data JPA = convenience layer (auto-generates boilerplate)
- H2 = the database engine

## Key Annotations on `Todo`

```java
@Entity                              // Hibernate: map this class to a DB table
@Table(name = "todos")               // explicitly name the table

@Id                                  // this field is the primary key
@GeneratedValue(strategy = IDENTITY) // DB auto-increments the id on every INSERT

@Column(nullable = false, columnDefinition = "TEXT")   // NOT NULL constraint
@Column(columnDefinition = "TEXT")                     // nullable (default)
@Column(nullable = false)                              // NOT NULL
```

The SQL Hibernate generates from these annotations:
```sql
create table todos (
    completed boolean not null,
    id        bigint generated by default as identity,
    description TEXT,
    title     TEXT not null,
    primary key (id)
)
```

## JpaRepository ‚Äî What You Get for Free

```java
public interface TodoRepository extends JpaRepository<Todo, Long> {}
// That's it. Spring generates:
```

| Method | SQL |
|---|---|
| `findAll()` | `SELECT * FROM todos` |
| `findById(id)` | `SELECT * FROM todos WHERE id = ?` ‚Üí `Optional<Todo>` |
| `save(todo)` | `INSERT` if id null, `UPDATE` if id present |
| `existsById(id)` | `SELECT COUNT(*) > 0 WHERE id = ?` |
| `deleteById(id)` | `DELETE FROM todos WHERE id = ?` |
| `count()` | `SELECT COUNT(*) FROM todos` |

No SQL. No implementation class. Spring generates a proxy at runtime.

## @Transactional

```java
@Transactional               // class level: every method runs in a transaction
@Transactional(readOnly=true) // method level: skips Hibernate dirty-checking ‚Üí faster
```

**What a transaction does:**
- All DB operations in the method = one atomic unit
- If ANY step throws RuntimeException ‚Üí all changes roll back
- If all steps succeed ‚Üí changes commit at the end

**readOnly = true:**
- Tells Hibernate "this method won't modify data"
- Hibernate skips dirty-checking (scan for changed fields) ‚Üí performance gain
- Always use on SELECT-only methods (getAllTodos, getTodoById)

## save() ‚Äî INSERT vs UPDATE Decision

```java
// id == null ‚Üí Hibernate generates INSERT (new row)
Todo todo = Todo.builder().title("Buy milk").completed(false).build();
todoRepository.save(todo); // INSERT INTO todos (title, completed) VALUES (?, ?)

// id != null ‚Üí Hibernate generates UPDATE (existing row)
Todo todo = Todo.builder().id(3L).title("Updated").completed(true).build();
todoRepository.save(todo); // UPDATE todos SET title=?, completed=? WHERE id=?
```

This is why the mapper passes the URL-path `id` explicitly for updates ‚Äî
so `save()` knows to UPDATE and not INSERT.

## Interview Questions & Answers

**Q: What is ORM?**
Object-Relational Mapper. Bridges Java objects to DB tables.
Class = Table, Object = Row, Field = Column.
You call `repository.save(todo)`, Hibernate generates the SQL.

**Q: What is the difference between JPA and Hibernate?**
JPA is a specification (an interface/standard defined by Jakarta EE).
Hibernate is a specific implementation of that spec.
Other implementations exist (EclipseLink, OpenJPA) but Hibernate is dominant.

**Q: What does Spring Data JPA add on top of JPA?**
It generates repository implementations at runtime via dynamic proxies.
You extend `JpaRepository`, declare no implementation, and Spring generates
`findAll()`, `findById()`, `save()`, `deleteById()` etc. automatically.

**Q: What does `@Transactional` do?**
Wraps the method in a database transaction. If a RuntimeException is thrown,
all DB changes in that method are rolled back. If the method completes normally,
changes are committed. Ensures atomicity ‚Äî all-or-nothing.

**Q: What is the difference between `@Transactional` and `@Transactional(readOnly=true)`?**
`readOnly=true` is a performance hint to Hibernate: skip dirty-checking.
Hibernate normally scans all loaded entities at end of transaction to detect
changes. With readOnly, it skips this scan since we guarantee no writes.

**Q: When does `save()` INSERT vs UPDATE?**
Spring Data checks if the entity's `@Id` field is null.
- null ‚Üí INSERT (new entity, DB assigns the id)
- not null ‚Üí UPDATE (merge/replace existing row)

**Q: What is `ddl-auto` and what value should you use in production?**
Controls how Hibernate manages the schema.
- `create-drop`: create on start, drop on stop (tests/dev)
- `validate`: verify schema matches entities, fail if not (safe for prod)
- `none`: do nothing (prod with Flyway/Liquibase handling migrations)
Never use `update` in production ‚Äî it can silently alter your schema.
